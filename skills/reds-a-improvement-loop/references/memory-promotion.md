# Memory Promotion Guide

Use this guide when deciding whether something belongs in daily memory, long-term memory, workspace rules, or nowhere.

## Put it in daily memory when it is

- a raw event from today
- a temporary task state
- an unresolved idea
- a note that may matter later but is not yet proven

Examples:
- "Tried ClawHub install; hit rate limits"
- "Danlin asked about capability evolution"
- "Need to retry sql-toolkit later"

## Put it in MEMORY.md when it is

- a stable user preference
- an ongoing project or durable goal
- a repeated lesson likely to matter across sessions
- an important decision with future consequences

Examples:
- "Danlin wants Reds A focused on profitable App Store app research and full-stack engineering"
- "Prefer bounded, practical improvement over autonomous self-modification"

## Put it in AGENTS.md when it changes how Reds A should operate

- recurring workflow rules
- lessons about when to ask vs act
- durable process improvements
- patterns that should guide future sessions

Examples:
- "Vet third-party skills before installing"
- "Prefer practical memory maintenance over meta-evolution"

## Put it in TOOLS.md when it is setup-specific

- local command quirks
- machine-specific paths or aliases
- environment gotchas
- preferred services or stack defaults

Never store secret values themselves.

## Do not promote when it is

- a one-off emotional beat
- overly abstract philosophy
- redundant with existing memory
- too vague to change future behavior
- likely to go stale quickly

## Promotion Checklist

Before promoting, ask:

1. Will this matter again in a week or a month?
2. Would future Reds A act differently because of this?
3. Is the wording short, concrete, and non-dramatic?
4. Does it belong in the chosen file more than anywhere else?
5. Am I preserving signal rather than clutter?

If two or more answers are "no," do not promote yet.
